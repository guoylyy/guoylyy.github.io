好的程序员，差的程序员

2015-10-22

本篇文章主要谈我对“好的程序员”和“差的程序员”的看法，是个人经验所谈，分歧在所难免，望看官们包涵。

（注：以下例子均来自我以往工作发生的真是场景，绝无编造之嫌）

好的程序员热衷产品和业务，坏的程序员只想接受任务

好的程序员明白，手头的技术是一盘散沙，是没有生命的物品，而自己是把这些无生命的物品与真实世界串联的手艺人，如果不能理解自己为什么使用技术，那么最好的做法是不要开始。所以，好的程序员理解业务、对业务感兴趣、为客户（包括产品经理）思考，帮助他们逾越技术鸿沟，达成真是价值。

坏的程序员只接受派发到手头的任务，对于背后的需求逻辑合理性、自己所做工作的业务场景不做甚解，并且以此作为逃避责任的接口。

好的程序员累积经验，坏的程序员累积毛病

好的程序员在工作中坚持采用长久性的解决方案，考虑软件长期的可维护性、效率、扩展性等诸多方面，把长期软件成本降低到最低，并把这些经验发扬下去，在以后的职业生涯中不断累积这方面的经验。

坏的程序员累积工作中不断采用临时方案解决问题，对软件开发中的诟病听之任之，没有想过试图改变这种糟糕的状况，在以后职业生涯中，不断累积毛病。

（直接结果[非常严重]：有些人累积了5年工作经验，有些人累积了5年工作毛病）

好的程序员注重沟通，坏的程序员善于抱怨

好的程序员认为问题都是可以通过沟通解决的，不断试图掌握一定的需求和业务的话语权，不断反馈进度报告，通过沟通让合作顺畅，进度保证；

坏的程序员一旦出现需求或者业务等问题，停留在原地等待，拒绝沟通交流，并且不认为这是自己工作的一部分。

and so on?

（我想到会不断添加的)

我坚持认为，其实好与不好，就是一念之间，不断坚持原则，坚决高要求，是不断奋斗的目标。

Plan of Github 100 Streaks

2015-09-23

说明

本计划是实现个人技术一次长期性提升而制定的计划，旨在帮助开发人员养成编码的良好习惯 、提升编码热情、锻炼编码技艺。 因此，本计划参与人员必须严格遵守一下规定，否则计划终止。

规定

powered by 177days of github

他提出的原则有：

  1. Edits to README don’t count unless significant in nature.
  2. No scripting; you must personally make the contributions.
  3. No funny business editing commit times and such.

这里总结和补充一下原则：

  1. 必须是有意义的提交，feature or fix，feature可以拆分，但是必须完整，可以测试。

  2. 不能隔天提交，中断即挑战失败；

  3. 不限制维护的项目，但是每一个项目必须有良好的 readme 说明，并且定期更新；

  4. 每日提交后记录自己提交的内容，记录在 commit 信息里，需要详细可阅读；

  5. 定期和伙伴 code review；

  6. 维护的代码持续集成、自动测试、报告、周知。

你将获得

  • 持续编码的良好习惯；
  • 维护一个完整项目的经历；
  • 代码编码能力和习惯（code review）

总之，如果和我一起完成了这项挑战，你一定是一个优秀的程序员。

我的爱情观

2015-08-29

我的爱情观

本文主要以一个“我”的角度谈谈对爱情的理解，不一定是正确的，也不一定是为大部分人所认同的，只以为我和认同我的人为准。

什么是我认为好的爱情

好的爱情是不断变化的

难度系数: 5

好的爱情不是一成不变的爱情。情感善变，烟花易冷，顷刻之间，咫尺西天，即使同一天早晚，情感都皆可折返。所以，好的爱情应该是合作开放地，接受任何变化，接受二人关系的变化，接受生活生计的变化，甚至接受如同“不思量，自难忘”的阴阳间隔的变化… 有自信接受好的，有勇气接受坏的，作为我，希望能做到这一点。

好的爱情是基于性吸引的魅力的

难度系数: 4

我时常在想，如果对于我的另一半，没有任何一点让我能够激起两人一起疯狂的冲动，那么这应该不是一种好的爱情吧。 对于我来说，基于性的吸引几乎占据了引发爱情的半壁江山，不管共赴巫山云雨还是踏遍小桥流水，都同样保持内心的悸动是一件非常好的事情啊。 同样，这也是对我来说的。

好的爱情是共同心智成熟

难度系数: 8

好的爱情，是共建心智的社区，这和我爱你不同，我爱你是和对方没有任何关系的一种连接，而共同成熟是两者能够在这段关系中提升自己的心智，磨练自己的意志，达到成为一个更好的人的目的。
所以，一定不要在以为时机成熟过后再开始，也不要以自己不成熟作为退却的接口，我眼中的好的爱情，是以60分的心态，80分的实力，一起交出两份都是 90 分的答卷。

好的爱情是看着对方的眼睛，却看到了全世界

难度系数: 9

当看着对方的眼睛的时候，如果我看到的不只是对方，而是整个世界，那么，这就是我认为的好的爱情。 这不是情话，能够让对方看到整个世界，这就是一种给予对方自由的力量，是一种让对自我的极度自信，是一种接受变化包容开放的心态…让对方看到未来的无限可能，本身就是一种很值得期待的事情，所以，我也希望让对方从自己眼中看到一个很大的世界。

RAML - 写项目文档也能有快感

2015-04-23

去年开始，开始在项目中大量使用前后端分离的架构，即后台大量使用 rest api，这样一来，一个项目的前后台编写就分离了，通常是两个人或者两拨人分别负责。 那么随之而来的问题也是比较明显的 —— 前后端的团队交流接口存在了天然的障碍。 由于我们是小团队，项目也充满着变化和不确定性，这样的交流障碍阻碍了项目的进展。

最初的 Rest API 文档解决方案

为了解决以上所述的问题，我们起初使用 markdown 来编写后端接口文档，然后由前端工程师来根据文档说明使用 api，其中 markdown 文档遵循了同意的模板规则，如：

1.1 xxx 接口
    method：post
    url： /login
    request:
        {
        ...
        }
    repsones:
       {
       ...
       }

这样，总算有了相应的解决方案。

优化的契机 —— RAML

而任何的解决方案都存在着相应的问题，用了一段时间过后，我们发现这样的方法有一些弊端：

  1. 规范性差，不同人写出来的格式不相同，造成了阅读困难；
  2. 难以查找，想要查找一个 API， 非常麻烦，需要在 Markdown 文档搜索，这对于 API 多一点来说有点麻烦；
  3. 格式编辑困难，只能纯手写，无专业文档工具

而在这个时候，我无意中看到了 RAML（Rest API Modeling Language），一时之间惊为神器，试用了一下，发现非常适合我们这个前后端文档编写的需求。

简单说来，RAML 定义了一个 Rest API 的文档编写规范，把 Rest 文档标准化了。 标准化带来的是规范，就是大家写的 Rest 文档都一样了，你好我好大家好，就像 xml 一样，可以用程序处理了。这样一来带来了一些好处：

  1. 我们可以利用工具解析 RAML 文档，生成我们想要的格式，比如组织很好的 html；
  2. 我们编写文档的时候可以有提示工具了（因为标准化了嘛）
  3. 我们阅读的时候更加高效快速无痛苦（还是因为标准化）。
